# Undertow: A Synthetic Benchmark Framework for Programmatic Tool Calling

*A small, controllable benchmark-generation framework for studying whether programmable workspaces make tool-using agents more reliable in hidden, stateful environments.*


## TL;DR

Undertow should be read less as a finished benchmark and more as a synthetic benchmark framework: a generator, protocol, and trace-auditing apparatus for constructing hidden stateful tool-use tasks under controlled difficulty knobs.

The question is narrow by design. Programmatic tool calling should not matter much when the task is a short list of isolated API calls. It should matter when the model must maintain state, build memory, verify progress, recover from failed actions, and control a partially observed environment over many steps. Undertow tests exactly that claim.

The current version has two task families. `hidden_dag` tests hidden dataflow: the model must compute a target value by discovering and executing the right sequence of generated functions. `repair_chain` tests hidden mutable state: the model must inspect tools, scan machines, track observation versions, apply repairs in the right order, and submit the repaired objects. Both families can be generated with controlled difficulty, both expose only primitive public tools, and both are scored by a hidden checker.

The first diagnostic run is useful, but it is not yet the final comparative claim. PTC agents solved `hidden_dag` at high reliability and made substantial progress on `repair_chain`; however, the initial standard-vs-PTC comparison exposed a harness state-serialization bug in standard mode. That bug has since been fixed, so the next clean run should replace the diagnostic tables.

The early lesson is therefore not "PTC wins." The better claim is: Undertow is already doing what a trace-first synthetic framework should do. It makes state-tracking failures visible, separates interface errors from world-model errors, and gives us a controlled way to ask when a programmable workspace becomes a genuine agentic advantage.

<a id="flow-framework"></a>
## Why this should be a framework, not just a benchmark

There is a temptation, especially when results begin to look interesting, to present a system like Undertow as a benchmark. That is not wrong, but it is incomplete. A single benchmark is a fixed measuring stick. Undertow is closer to a small machine for making measuring sticks.

The distinction matters.

A fixed benchmark gives us a score. A benchmark framework gives us a laboratory. It lets us vary the number of hidden dependencies, the depth of the dependency chain, the number of distractors, the amount of partial observability, the rate of stale observations, the strictness of the budget, and the interface through which the agent acts. It also lets us inspect not only whether the model reached the answer, but how it failed when it did not.

That is the right framing for Undertow. The point is not to claim that two synthetic task families exhaust the agent problem. They do not. The point is to build a controlled setting where the difference between ordinary tool calling and programmatic tool calling can be isolated rather than merely asserted.

The central question is:

> When the external world is hidden, stateful, and only accessible through primitive tools, does a programmable workspace make the model a more reliable controller?

That question cannot be answered cleanly with a single hand-written task. It needs generation, knobs, traces, repeatable seeds, failure labels, and rerunnable experiments. Undertow is the beginning of that framework.

<a id="flow-question"></a>
## The question Undertow is built to ask

The broad conversation about agents often bundles several capabilities together: tool selection, code generation, planning, memory, search, recovery, verification, and final answer formatting. Bundling them is natural in product settings, but it is a problem for evaluation. When an agent fails, we need to know what failed.

Did it choose the wrong tool? Did it call the right tool with the wrong arguments? Did it forget a value it had already discovered? Did it reuse an observation after the world had changed? Did it complete the hidden state transition but submit the answer in the wrong shape? These are different failures. A serious benchmark framework should not collapse them into one number.

Undertow isolates a specific capability: the ability to build and use a local control layer around external tools.

A normal tool-calling agent receives a tool schema, makes a call, reads the observation, and decides what to do next through the conversation loop. A programmatic tool-calling agent receives the same public tools, but it can also write code around them. It can build ledgers, queues, caches, planners, parsers, verifiers, retry policies, and small domain-specific solvers. The hidden state remains hidden. The reward checker remains protected. The only difference is that the model can turn repeated tool use into a program.

That difference is the object of study.

<a id="flow-funcbenchgen"></a>
## The FuncBenchGen lesson

FuncBenchGen is the nearest intellectual reference point. It frames multi-step function calling as traversal over a hidden function-dependency DAG. The model receives generated functions, input variables, and a target variable, but not the graph itself. It must infer the correct call sequence from the public function schemas and compute the target value.

The important idea is not merely "use a DAG." The important idea is experimental discipline. FuncBenchGen is synthetic, contamination-free, and controllable. It can vary required function count, dependency depth, irrelevant function count, and whether distractors are connected to the solution path. Because the tasks are generated, the benchmark is less exposed to memorization. Because the graph structure is controlled, the benchmark can say which difficulty factor caused the degradation.

That is the posture Undertow should borrow.

Undertow begins where FuncBenchGen is strongest: hidden dataflow. Its `hidden_dag` family is deliberately close to that setup. A model must track known values, call functions whose inputs are available, avoid distractors, and submit the target value. This gives Undertow an anchor: a clean dataflow task whose ideal PTC solution is a small topological executor.

But Undertow then moves one step further. It asks what happens when the task is not only hidden dataflow, but hidden mutable state.

In `repair_chain`, the world changes. Machines move through repair stages. Observation versions become stale. Tools are compatible only with certain parts and stages. The agent's location matters. The task is no longer only "compute the missing variable." It is "operate a partially observed state machine until the hidden checker accepts the final world."

This is where programmatic tool calling becomes most natural. A local workspace is not just a convenience; it is the place where the model can maintain the operating record that the environment itself will not hand it for free.

<a id="flow-components"></a>
## Undertow as a synthetic benchmark framework

Undertow is built from four components.

**First**, it has task generators. A generator samples a hidden world from explicit parameters. For `hidden_dag`, those parameters include the number of required functions, dependency depth, connected distractors, disconnected distractors, and seeds. For `repair_chain`, they include the number of machines, repair steps, tools, distractor tools, staleness behavior, partial observation behavior, and seeds.

**Second**, it has a public interface. The agent never receives the hidden state directly. It receives tool schemas, public identifiers, and a target description. All contact with the world must happen through primitive tools.

**Third**, it has multiple interaction modes. The standard mode exposes native tools through the usual tool-call loop. The PTC mode exposes a persistent Python workspace and a small `ptc` package that routes calls to the same primitive tools. The tools are not stronger in PTC mode. The model simply gets a programmable surface from which to organize them.

**Fourth**, it has trace-level auditing. Every call records the tool name, arguments, observation, state delta, consumed budget, and failure code. This lets Undertow report success, but also progress, wasted calls, repeated calls, stale references, incorrect values, distractor actions, premature submissions, and budget exhaustion.

That last point is central. A benchmark framework should produce evidence, not just a leaderboard cell.

<a id="flow-formalization"></a>
## Formalization

An Undertow task is a tuple:

$$
\tau = (S_0, \Omega, A, T, R, B)
$$

where:

* $S_0$ is the hidden initial state.
* $\Omega$ is the set of observation functions.
* $A$ is the set of state-changing actions.
* $T$ is the target condition.
* $R$ is the hidden reward checker.
* $B$ is the tool-call budget.

The agent does not observe $S_0$ directly. It receives a public task specification containing tool schemas, public identifiers, task parameters, and a target description.

At step $t$, the agent chooses an operation:

$$
a_t \in \Omega \cup A
$$

The environment applies the transition:

$$
S_{t+1} = \mathrm{transition}(S_t, a_t)
$$

and returns a public observation:

$$
o_t = \mathrm{observe}(S_{t+1}, a_t)
$$

The episode ends when the agent submits an answer or exhausts the budget. Success is defined only by the hidden checker:

$$
R(S_T, \mathrm{answer}) = 1
$$

This is a strict contract. The agent may write a local verifier, maintain a ledger, or decide that it is done. None of those local beliefs are authoritative. The environment decides.

<a id="flow-modes"></a>
## The two interaction modes

The comparison is meaningful only if both modes solve the same generated tasks against the same hidden checker.

### Standard mode

In standard mode, the model receives the public task specification and native tools. For `hidden_dag`, those tools are generated functions plus `submit`. For `repair_chain`, the tools include operations such as:

* `scan_object`
* `inspect_tool`
* `move_to`
* `apply_tool`
* `test_object`
* `submit`

The model acts through the ordinary conversation/tool-call loop. It must remember what it learned, choose the next tool, and eventually submit an answer.

### PTC mode

In PTC mode, the model receives a persistent Python workspace. A small package exposes the public task and tools:

```python
from ptc import task, tools, finish
```

The agent can inspect the public spec:

```python
task.spec()
```

It can list and call the same primitive tools:

```python
tools.list()
tools.call("scan_object", object_id="machine_0")
tools.scan_object(object_id="machine_0")
```

It submits with:

```python
finish({"repaired": ["machine_0", "machine_1"]})
```

The PTC package is not an oracle. It does not expose hidden state. It does not expose reward. It is only a programmatic interface to the same environment.

The hypothesis is that this interface changes what the model can reliably build: not new facts, but better machinery.

<a id="flow-hidden-dag"></a>
## Task family I: hidden_dag

`hidden_dag` is Undertow's dataflow family. It is intentionally close to the FuncBenchGen style of task.

The generator creates deterministic functions over integer variables. The public spec includes initial variable values, generated function names, input variables, output variables, JSON schemas, and a target variable. The hidden state contains the true value table and the ground-truth solution order.

A small public spec might look like this:

```json
{
  "family": "hidden_dag",
  "initial_values": {
    "v0": 964,
    "source_5": 494
  },
  "target": "v5",
  "functions": [
    {
      "name": "core_0",
      "inputs": ["v0"],
      "output": "v1",
      "description": "Transforms v0 into v1."
    },
    {
      "name": "core_4",
      "inputs": ["v4", "v6", "v8"],
      "output": "v5",
      "description": "Combines v4, v6, v8 into v5."
    }
  ]
}
```

A successful function call returns the newly established variable and value:

```json
{
  "ok": true,
  "variable": "v1",
  "value": 6761
}
```

The agent must call functions only when their inputs are known, store returned values under the right variable names, avoid distractors, and submit the target value after it has been computed.

The main difficulty knobs are:

* `n_core`: number of required functions.
* `depth`: longest required dependency path.
* `n_connected_distractors`: irrelevant functions connected to known or core variables.
* `n_disconnected_distractors`: irrelevant functions outside the solution path.
* `observation_mode`: whether observations merely return the latest value or restate known values.

The ideal PTC solution is simple and revealing. Parse the function list, keep a `known_values` dictionary, repeatedly find callable functions, call them, cache their outputs, and submit the target. This is not a profound algorithm. It is a small topological executor. But that is precisely the point: PTC can turn a multi-turn state-tracking problem into a local execution loop.

<a id="flow-repair-chain"></a>
## Task family II: repair_chain

`repair_chain` is Undertow's stateful family.

The world contains broken machines, tools, locations, and hidden repair stages. A machine has fields like:

```json
{
  "id": "machine_0",
  "status": "broken",
  "part": "part_0",
  "location": "bay_1",
  "repair_stage": 0,
  "observation_version": 0
}
```

The public spec lists machine ids, bay ids, tool ids, and operation schemas. The agent can call:

* `scan_object(object_id)`
* `inspect_tool(tool_id)`
* `move_to(location)`
* `apply_tool(tool_id, object_id, observation_version)`
* `test_object(object_id, observation_version)`
* `submit(answer)`

Tool inspection makes the task inferable:

```json
{
  "ok": true,
  "tool_id": "tool_3",
  "kind": "repair",
  "compatible_parts": ["part_0", "part_2"],
  "compatible_stages_by_part": {
    "part_0": [1],
    "part_2": [0, 2]
  }
}
```

A competent solver can inspect tools, scan machines, build a map from `(part, stage)` to `tool_id`, move to each machine, apply the correct stage tools, update observation versions after every state-changing action, test repaired machines, and submit the repaired ids.

The main difficulty knobs are:

* `n_objects`: number of machines to repair.
* `n_repair_steps`: sequential repair stages per machine.
* `n_tools`: number of real tools.
* `n_distractor_tools`: number of useless tools.
* `staleness_rate`: whether object observations become stale after repair actions.
* `partial_observation_rate`: whether scans hide fields such as part or location.

This task is not just a DAG of values. It is a mutable state machine. A standard tool-calling agent has to keep several tables in conversation: which machine is in which bay, which part it has, which tool repairs which stage, what the current stage is, which observation version is current, where the agent is located, and which machines have already been tested.

The PTC agent can instead build a repair ledger:

```python
machines = {}
stage_tool = {}
current_location = None
completed = set()
```

It can update that ledger after every observation. It can assert that a tool exists for the current `(part, stage)` pair. It can refresh stale observations. It can group repairs by location. It can stop retrying after a constraint violation.

This is the kind of problem Undertow is for: not whether Python is magic, but whether the model can use code as an operating record for a hidden world.

<a id="flow-metrics"></a>
## Metrics and failure taxonomy

Undertow reports both outcome and shape of behavior.

The core metrics are:

* `success`: whether the final submitted answer satisfies the hidden checker.
* `num_calls`: consumed tool-call budget.
* `budget`: total budget.
* `remaining_budget`: unused calls at the end.
* `budget_used_ratio`: consumed calls divided by budget.
* `progress_score`: family-specific normalized partial progress.
* `useful_calls`: consumed calls with a non-empty state delta.
* `wasted_calls`: consumed calls that produced a failure code.
* `failure_counter`: counts by machine-checked failure type.
* `first_failure`: first failure code in the trace.
* `final_failure`: final failure code for failed episodes.

The failure codes are deliberately mechanical:

* `invalid_action`: the tool does not exist or the episode is already closed.
* `schema_error`: arguments do not match the schema.
* `unknown_state_reference`: the agent referenced an unknown object, tool, or variable.
* `incorrect_value`: the agent supplied a known variable with the wrong value.
* `stale_state_reference`: the agent used an obsolete observation version.
* `violated_constraint`: the action was syntactically valid but violated world constraints.
* `distractor_action`: the agent used an irrelevant function or distractor tool.
* `premature_submit`: the submitted answer did not satisfy the target.
* `budget_exceeded`: the agent tried to continue after consuming its budget.
* `repeated_call`: the agent repeated an identical call.

<a id="flow-grid"></a>
## Experimental grid

Undertow v0 uses a small fixed grid. The goal is not exhaustive coverage; the goal is a clean first cut that is inspectable by hand and still rich enough to expose mode differences.

```
hidden_dag:
  n_core in {5, 10}
  depth in {2, 5}
  n_connected_distractors in {0, 5}
repair_chain:
  n_objects in {1, 3}
  n_repair_steps in {2, 3}
  n_distractor_tools in {0, 2}
seeds:
  0..4
modes:
  standard, ptc
```

The budget is set to three times the public-solver call count.

For `hidden_dag`, the public-solver count is all generated functions plus submit. For `repair_chain`, it is inspecting every tool once, scanning every object once, moving as needed, applying every required repair stage, testing every object, and submitting.

The multiplier matters. A budget that fits only an oracle-perfect trace mostly measures whether the model guessed the intended path immediately. Undertow should allow discovery, wrong hypotheses, repeated scans, and occasional recovery while still making waste visible.

<a id="flow-diagnostic"></a>
## First diagnostic run

The first run compared standard and PTC modes across the generated grid using Qwen3.6-35B-A3B and GLM-5. The normalized result set contains 307 episodes, with most cells containing 40 episodes. One PTC `repair_chain` slice for GLM-5 is incomplete and should be treated as directional.

The headline pattern was strong for PTC. On `hidden_dag`, PTC agents built the expected local machinery: a value table, a queue of callable functions, and a simple execution loop. They solved the task at high reliability. On `repair_chain`, PTC agents often completed the hidden repair transitions, but many failed the final reward because they submitted prose or an incorrectly shaped final answer rather than the required structured object.

That is a real interface failure, but it is not the same as failing to operate the hidden world. In a benchmark framework, the distinction matters.

The more important event was that the standard comparison exposed a harness bug. Standard mode reconstructed the environment from a serialized snapshot between native tool calls, but the snapshot preserved only generic episode bookkeeping. It did not preserve family-specific mutable hidden state, such as the known value table in `hidden_dag` or the current location and repair stages in `repair_chain`.

PTC mostly avoided this issue because its primitive tools were served through a live RPC-backed session. Standard mode did not. A standard agent could compute `v1`, then make the next call in an environment that had forgotten `v1`. It could move to `bay_0`, then apply a tool in an environment whose `current_location` had reset.

That means the pre-fix standard-vs-PTC numbers are diagnostic, not publication-grade.

<a id="flow-diagnostic-lessons"></a>
## What the diagnostic run still shows

The diagnostic run is not useless just because the standard harness had a bug. In fact, it shows why trace-first evaluation matters.

**First**, PTC agents appear to convert `hidden_dag` into the intended local-state problem. The best traces look less like improvised conversation and more like small executors. They parse the public spec, maintain known values, call functions when inputs are available, and submit the target.

**Second**, `repair_chain` shows a useful separation between progress and reward. Several PTC agents made the right state transitions but failed final submission formatting. Reward is the legal judgment of the benchmark. Progress is the audit trail. A benchmark framework should preserve both.

**Third**, the standard traces revealed a plausible behavioral contrast: standard agents often responded to uncertainty by repeating tools, refreshing observations, or trying nearby schemas rather than maintaining an explicit ledger. That pattern is exactly what Undertow is meant to measure after the harness fix.

**Fourth**, the harness bug itself is an evaluation result of a different kind. It shows that hidden mutable state is easy to mishandle, not only for models but for benchmark infrastructure. A trace-audited framework can catch that before the bug hardens into a leaderboard claim.

<a id="flow-hidden-dag-results"></a>
## hidden_dag results, cautiously stated

On `hidden_dag`, the PTC agents did what the framework was designed to make possible. They treated the task as a dataflow execution problem.

The cleanest interpretation is:

* The local state machine for `hidden_dag` is a table of known values plus a queue of callable functions.
* PTC agents can build that state machine naturally inside the workspace.
* Standard agents may also solve the task, but they are more exposed to conversational state loss and repeated-call recovery loops.
* The pre-fix standard results should be rerun before any final mode comparison is made.

<a id="flow-repair-chain-results"></a>
## repair_chain results, cautiously stated

`repair_chain` is the more important family for Undertow's long-term purpose because it stresses mutable state.

The current result should be described carefully. PTC agents often completed the hidden repair process but failed at the final answer contract. That means the framework successfully separated two failure types: operating the world and satisfying the submission interface.

This distinction should be reflected directly in the essay. A final answer-shape error is not harmless. In deployment, an interface error can still break the system. But for benchmark design, it is diagnostically useful. It tells us that the next fix is not necessarily "make the model better at repair," but "make the finish contract explicit and test whether the repaired state is being submitted in the expected shape."

<a id="flow-ptc"></a>
## Why this is interesting for PTC

The strongest version of the PTC claim is not that a model with Python is always better. That would be too broad and probably false.

The stronger claim is conditional:

> PTC should help when the task can be converted into a local state machine whose updates depend on external tools.

In `hidden_dag`, the state machine is a value table. In `repair_chain`, it is a repair ledger. In both cases, the workspace lets the model write down the operational reality of the task and update it after each call.

This is where PTC differs from ordinary tool use. The model is not merely choosing the next action. It is building the apparatus that chooses actions: a parser, a cache, a planner, a checker, a retry policy, and a final submission constructor.

That is also why Undertow should remain synthetic for now. Synthetic tasks let us vary exactly the dimensions that should matter to this claim:

* longer dependency depth;
* more connected distractors;
* more objects;
* more repair stages;
* higher staleness rate;
* stricter budgets;
* noisier observations;
* more brittle final contracts.

A realistic benchmark can tell us whether an agent works in a particular world. A synthetic framework can tell us which property of the world broke it.

<a id="flow-next"></a>
## What to fix next

The next version should make three changes before stronger claims are made.

**First**, rerun the standard comparison after the snapshot fix. Family-specific mutable hidden state must be serialized and restored correctly in both task families. Without that, the mode comparison is confounded.

**Second**, make the final answer contract explicit in the PTC prompt and public task spec. The prompt should show the required final shapes:

`finish({"value": 1234})`

for `hidden_dag`, and:

`finish({"repaired": ["machine_0", "machine_1"]})`

for `repair_chain`.

<a id="flow-not"></a>
## What Undertow is not

Undertow is not a claim that synthetic tasks are sufficient for agent evaluation. They are not. Real environments have messy semantics, long tails, human constraints, and objective functions that are harder to formalize.

Undertow is not a claim that PTC always beats standard tool calling. It should not. If a task is short, stateless, and obvious, a programmable workspace may add little or even introduce overhead.

Undertow is not a benchmark of coding ability alone. The hidden state remains outside the workspace. The model cannot solve the task by local computation alone. It must use the environment tools, and the reward checker remains external.

The snapshot fix and clean rerun are necessary before the standard-vs-PTC comparison should be treated as a result.

<a id="flow-closing"></a>
## Closing

The old habit in tool-use benchmarks is to ask whether the model reached the answer. Undertow keeps that question, but it is more interested in the account book: what the agent knew, what it changed, what it repeated, what it used too early, what it forgot, and what it submitted.

That is why the right framing is a synthetic benchmark framework.

A benchmark gives a number. A framework gives a controlled way to generate worlds, vary difficulty, audit traces, and locate failure. For programmatic tool calling, that is the more valuable instrument.

Programmatic tool calling is interesting only when the program becomes more than syntax. It must become a ledger, a planner, a verifier, a memory, and a disciplined interface to a world the model cannot inspect directly.

Undertow is a small framework for seeing when that happens.

The cautious conclusion is enough for now: PTC appears promising when hidden tool-use tasks can be turned into explicit local state machines, but the clean comparative claim should wait for the fixed rerun. That is not a weakness of the project. It is the reason to build the framework in the first place.
