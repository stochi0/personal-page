# Undertow: A Synthetic Benchmark Experiment for Programmatic Tool Calling

*A small, controllable benchmark experiment for studying whether programmable workspaces make tool-using agents more reliable in hidden, stateful environments.*


## TL;DR

Programmatic tool calling should not matter much when the task is a short list of isolated API calls. It should matter when the model must maintain state, build memory, verify progress, recover from failed actions, and control a partially observed environment over many steps. Undertow tests exactly that claim.

The current version has two task families. `hidden_dag` tests hidden dataflow: the model must compute a target value by discovering and executing the right sequence of generated functions. `repair_chain` tests hidden mutable state: the model must inspect tools, scan machines, track observation versions, apply repairs in the right order, and submit the repaired objects. Both families can be generated with controlled difficulty, both expose only primitive public tools, and both are scored by a hidden checker.

The first diagnostic run is useful, but it is not yet the final comparative claim. PTC agents solved `hidden_dag` at high reliability and made substantial progress on `repair_chain`; however, the initial standard-vs-PTC comparison exposed a harness state-serialization bug in standard mode. That bug has since been fixed, so the next clean run should replace the diagnostic tables.

The early lesson is therefore not "PTC wins." The current experiment setup makes state-tracking failures visible, separates interface errors from world-model errors, and gives us a controlled way to ask when a programmable workspace becomes a genuine agentic advantage.

<a id="flow-why"></a>
## Why Undertow exists

Agent evaluations often bundle tool selection, planning, memory, recovery, verification, and final answer formatting into one score. That is natural in products, but it makes failures hard to understand. Did the model choose the wrong tool, forget a value, reuse stale state, complete the world transition but submit the wrong shape, or simply run out of budget?

Undertow isolates one question:

> When the external world is hidden, stateful, and only accessible through primitive tools, does a programmable workspace make the model a more reliable controller?

The point is not that two synthetic task families exhaust the agent problem. They do not. The point is to create a controlled setting with generators, knobs, traces, repeatable seeds, failure labels, and rerunnable experiments. Undertow lets us vary hidden dependencies, dependency depth, distractors, partial observability, stale observations, budget strictness, and the interface through which the agent acts.

FuncBenchGen is the nearest intellectual reference point. It frames multi-step function calling as traversal over a hidden function-dependency DAG: the model receives generated functions, input variables, and a target variable, but not the graph itself. The important lesson is experimental discipline. A synthetic benchmark can be contamination-free, controllable, and precise about which capability it is testing.

FuncBenchGen's world is mostly **functional**: functions consume values and produce values. The task is to compute a target variable by executing the correct sequence of generated functions. The paper explicitly casts tool use as traversal over a hidden function-dependency DAG, with controllable difficulty through graph size, dependency depth, and distractor functions.

Undertow's world is **environmental**: the agent interacts with a hidden state through tools. Observations are partial, actions can change the world, stale observations can become invalid, and the model must maintain an operational record of what is true now.

The deeper difference is this: FuncBenchGen evaluates multi-step tool-call sequencing. Undertow evaluates tool-mediated control under hidden mutable state, and asks whether PTC is the right interface for that control problem.

<a id="flow-ptc-change"></a>
## What PTC changes

A normal tool-calling agent receives a schema, makes a call, reads the observation, and decides what to do next through the conversation loop. A PTC agent receives the same public tools, but can write code around them: queues for callable work, ledgers for state, parsers for observations, retry policies, and final-answer constructors. The hidden state remains hidden. The reward checker remains protected. The only difference is the control surface.

The strongest version of the PTC claim is conditional:

> PTC should help when the task can be converted into a local state machine whose updates depend on external tools.

In `hidden_dag`, that state machine is a value table. In `repair_chain`, it is a repair ledger. The benchmark is not checking whether Python is magic. It is checking whether a model uses a programmable workspace to become a more reliable agent.

<a id="flow-setup"></a>
## Benchmark setup

Undertow has task generators, a public interface, two interaction modes, and a hidden checker. A generator samples a hidden world from explicit parameters. The agent never receives that hidden state directly; it receives public identifiers, tool schemas, and a target description. All contact with the world happens through primitive tools.

An Undertow task is a tuple:

$$
\tau = (S_0, \Omega, A, T, R, B)
$$

where:

* $S_0$ is the hidden initial state.
* $\Omega$ is the set of observation functions.
* $A$ is the set of state-changing actions.
* $T$ is the target condition.
* $R$ is the hidden reward checker.
* $B$ is the tool-call budget.

The agent does not observe $S_0$ directly. It receives a public task specification containing tool schemas, public identifiers, task parameters, and a target description.

At step $t$, the agent chooses an operation:

$$
a_t \in \Omega \cup A
$$

The environment applies the transition:

$$
S_{t+1} = \mathrm{transition}(S_t, a_t)
$$

and returns a public observation:

$$
o_t = \mathrm{observe}(S_{t+1}, a_t)
$$

The episode ends when the agent submits an answer or exhausts the budget. Success is defined only by the hidden checker:

$$
R(S_T, \mathrm{answer}) = 1
$$

This is a strict contract. The agent may write a local verifier, maintain a ledger, or decide that it is done. None of those local beliefs are authoritative. The environment decides.

### Interaction modes

**Standard mode** exposes native tools through the usual tool-call loop. For `hidden_dag`, those tools are generated functions plus `submit`. For `repair_chain`, the tools include:

* `scan_object`
* `inspect_tool`
* `move_to`
* `apply_tool`
* `test_object`
* `submit`

The model must remember what it learned, choose the next tool, and eventually submit an answer.

**PTC mode** exposes a persistent Python workspace. A small package routes calls to the same primitive tools:

```python
from ptc import task, tools, finish
```

The agent can inspect the public spec:

```python
task.spec()
```

It can list and call the same primitive tools:

```python
tools.list()
tools.call("scan_object", object_id="machine_0")
tools.scan_object(object_id="machine_0")
```

It submits with:

```python
finish({"repaired": ["machine_0", "machine_1"]})
```

The PTC package is not an oracle. It does not expose hidden state. It does not expose reward. It is only a programmatic interface to the same environment.

## Task family I: hidden_dag

It asks: "Given a hidden dataflow problem shaped like a DAG, does a persistent programmable workspace make the model more reliable than ordinary tool calling?"

The generator creates a set of deterministic functions over integer variables. The public spec includes:

* **initial variable values**
* **target variable**
* **generated function names**
* **each function's input variables**
* **each function's output variable**
* **JSON tool schemas**

The hidden state contains the true value table and the ground-truth solution order. The model does not see that table. It must produce values by calling functions with correct already-known inputs.

A small public spec might look like this (`seed=3`, `n_core=5`, `depth=3`):

```json
{
  "family": "hidden_dag",
  "initial_values": {
    "v0": 343,
    "source_3": 706,
    "source_4": 478
  },
  "target": "v3",
  "functions": [
    {
      "name": "core_0",
      "inputs": ["v0"],
      "output": "v1",
      "description": "Transforms v0 into v1."
    },
    {
      "name": "core_3",
      "inputs": ["source_3"],
      "output": "v4",
      "description": "Transforms source_3 into v4."
    },
    {
      "name": "core_4",
      "inputs": ["source_4"],
      "output": "v5",
      "description": "Transforms source_4 into v5."
    },
    {
      "name": "core_1",
      "inputs": ["v1", "v4"],
      "output": "v2",
      "description": "Combines v1, v4 into v2."
    },
    {
      "name": "core_2",
      "inputs": ["v2", "v5"],
      "output": "v3",
      "description": "Combines v2, v5 into v3."
    },
    {
      "name": "connected_distractor_0",
      "inputs": ["v2"],
      "output": "d_connected_0",
      "description": "Produces a type-compatible but irrelevant value from v2."
    }
  ]
}
```

This is a dependency DAG: `v0 → v1`, parallel branches `source_3 → v4` and `source_4 → v5`, then merges `v1 + v4 → v2` and `v2 + v5 → v3`.

A successful function call returns:

```json
{
  "ok": true,
  "variable": "v1",
  "value": 596
}
```

The model keeps a ledger of known values, starting from `initial_values`. It repeatedly calls only functions whose inputs are known, stores each returned value under the returned variable name, and submits once the target appears.

The core algorithm is a topological executor:

```python
known = dict(task["initial_values"])
while target not in known:
    for fn in functions:
        if all(inp in known for inp in fn["inputs"]):
            result = call(fn["name"], **{inp: known[inp] for inp in fn["inputs"]})
            known[result["variable"]] = result["value"]
submit(value=known[target])
```

The challenge is dataflow discipline:

* **Callable inputs** — call only functions whose inputs are known
* **Exact values** — pass the exact known integer for each input, not a guess
* **Ledger updates** — extract the returned value and store it under the returned variable name
* **Distractor avoidance** — avoid connected distractors that consume budget without advancing the target
* **Target submit** — submit the target value only after computing it

</WritingDisclosure>

### What goes wrong

* **Inputs must match exactly.** The environment checks `kwargs[input] == true_value`. Calling `core_1(v1=596, v4=999)` fails with `incorrect_value` even if `v4` was computed earlier and the model mis-typed it.
* **Call order matters.** Calling `core_1` before `core_3` or `core_4` fails with `unknown_state_reference` because `v4` is not yet in the ledger.
* **Distractors are callable but wasteful.** `connected_distractor_0(v2=5630)` may return `ok: true` but is classified as a `distractor_action`.
* **Noisy mode** (used in medium and hard configs) can return a fake `{ok: true, value: ...}` on wrong inputs instead of an error, polluting the ledger with incorrect values.
* **Minimal observation mode** returns only the new variable per call, not a reminder of everything known. The model must track state itself across turns.

The difficulty knobs are:

* **`n_core`** — number of required functions
* **`depth`** — longest required dependency path
* **`n_connected_distractors`** — irrelevant functions connected to known or core variables
* **`n_disconnected_distractors`** — irrelevant functions outside the solution path
* **`observation_mode`** — whether the environment returns only the new observation or reminds the model of known values

The PTC angle is simple: writing a topological executor turns a multi-turn memory problem into a local state-management problem. Standard tool-calling agents can solve small DAGs too, but they are more exposed to context-level slips: using the whole observation object as an integer, calling a child before its parent, forgetting a returned value, or substituting an earlier value with a similar name. `hidden_dag` is Undertow's accounting ledger: exact, dry, and unforgiving.

## Task family II: repair_chain

`repair_chain` is the more stateful family. The model is not just computing a value. It is controlling a hidden state machine.

The world contains broken machines, tools, locations, and hidden repair stages. Each machine has fields like:

```json
{
  "id": "machine_0",
  "status": "broken",
  "part": "part_0",
  "location": "bay_1",
  "repair_stage": 0,
  "observation_version": 0
}
```

The public spec includes:

* **machine ids**
* **bay ids**
* **available tool ids**
* **operation schemas**

It does not reveal the hidden required-tool sequence per machine. A typical public spec looks like:

```json
{
  "objects": ["machine_0", "machine_1"],
  "locations": ["bay_0", "bay_1", "bay_2"],
  "tools_available": ["tool_0", "tool_1", "tool_2", "tool_3", "tool_4", "tool_5", "distractor_tool_0"],
  "target": "repair all critical machines"
}
```

The model can call:

* **`scan_object(object_id)`** — read machine state
* **`inspect_tool(tool_id)`** — discover compatible parts and stages
* **`move_to(location)`** — position the agent at a bay
* **`apply_tool(tool_id, object_id, observation_version)`** — advance repair stage
* **`test_object(object_id, observation_version)`** — confirm a machine is repaired
* **`submit(repaired=[...])`** — finish when all critical machines pass

The tool inspection response is what makes the task publicly inferable:

```json
{
  "ok": true,
  "tool_id": "tool_0",
  "kind": "repair",
  "compatible_parts": ["part_0", "part_1"],
  "compatible_stages_by_part": {
    "part_0": [0],
    "part_1": [0]
  }
}
```

From inspections across all tools, a solver builds a map from `(part, stage)` to `tool_id`. For `seed=12` with two machines and two repair steps, that map is:

* `(part_0, stage 0) → tool_0`
* `(part_0, stage 1) → tool_3`
* `(part_1, stage 0) → tool_0`
* `(part_1, stage 1) → tool_1`

<WritingDisclosure title="How a model could potentially solve it">

The solver first inspects every tool once, builds a `(part, stage) → tool_id` map, and skips distractors. Then it scans each machine, moves to its bay, applies the stage-appropriate tools, updates `observation_version` after every successful repair action, tests the object, and submits the repaired ids.

```python
move_to(location)
for stage in current_stage .. n_repair_steps-1:
    tool = stage_tool_map[(part, stage)]
    apply_tool(machine, tool, observation_version)
    observation_version = response.observation_version
test_object(machine, observation_version)
```

The `observation_version` update is the important state edge. Reusing an old version causes `stale_state_reference`.

</WritingDisclosure>

### What goes wrong

* **Wrong location.** `apply_tool` fails if the agent is not at the machine's bay.
* **Wrong tool for stage.** Tools must match the hidden sequence exactly; the wrong tool returns `violated_constraint`.
* **Stale observation_version.** After each repair step (when `staleness_rate > 0`), the version increments. Reusing an old version fails with `stale_state_reference`.
* **Distractor tools.** `distractor_tool_*` always fails with `distractor_action`.
* **Partial scans.** When `partial_observation_rate > 0`, `scan_object` may hide `part` or `location`, requiring extra discovery.
* **Premature submit.** Listing machines as repaired before finishing all stages fails with `premature_submit`.

The difficulty knobs are:

* **`n_objects`** — number of machines to repair
* **`n_repair_steps`** — number of sequential repair stages per machine
* **`n_tools`** — number of real tools
* **`n_distractor_tools`** — number of useless tools
* **`staleness_rate`** — whether object observations become stale after repair actions
* **`partial_observation_rate`** — whether scans hide fields such as part or location

This is where `repair_chain` stresses PTC. A standard tool-calling model has to keep several tables in its head:

* **machine → bay** — which machine is in which bay
* **machine → part** — which machine has which part
* **(part, stage) → tool** — which tool repairs which stage for that part
* **repair stage** — what the current repair stage is
* **observation version** — what the current observation version is
* **agent location** — where the agent is currently located
* **tested machines** — which machines have already been tested

Each successful repair action changes state. A PTC agent can write a small repair ledger:

```python
machines = {}
stage_tool = {}
current_location = None
```

That ledger can log calls, assert that a stage exists before applying a tool, update the observation version after every successful `apply_tool`, group machines by location, and stop retrying a tool after a constraint violation.

## Metrics and experimental grid

Undertow reports both outcome and shape of behavior.

The core metrics are:

* `success`: whether the final submitted answer satisfies the hidden checker.
* `num_calls`: consumed tool-call budget.
* `budget`: total budget.
* `remaining_budget`: unused calls at the end.
* `budget_used_ratio`: consumed calls divided by budget.
* `progress_score`: family-specific normalized partial progress.
* `useful_calls`: consumed calls with a non-empty state delta.
* `wasted_calls`: consumed calls that produced a failure code.
* `failure_counter`: counts by machine-checked failure type.
* `first_failure`: first failure code in the trace.
* `final_failure`: final failure code for failed episodes.

The failure codes are deliberately mechanical:

* `invalid_action`: the tool does not exist or the episode is already closed.
* `schema_error`: arguments do not match the schema.
* `unknown_state_reference`: the agent referenced an unknown object, tool, or variable.
* `incorrect_value`: the agent supplied a known variable with the wrong value.
* `stale_state_reference`: the agent used an obsolete observation version.
* `violated_constraint`: the action was syntactically valid but violated world constraints.
* `distractor_action`: the agent used an irrelevant function or distractor tool.
* `premature_submit`: the submitted answer did not satisfy the target.
* `budget_exceeded`: the agent tried to continue after consuming its budget.
* `repeated_call`: the agent repeated an identical call.

  <p><em>Failure codes by family and mode. The taxonomy separates interface errors, state-tracking errors, distractor actions, repeated calls, and budget exhaustion.</em></p>

This taxonomy is the difference between a score and an audit. Two agents may both fail, but one may fail before discovering the first useful fact while another may repair the world and then submit the wrong final shape. Those are not the same failure.

### Experimental grid

Undertow v0 uses a small fixed grid. The goal is not exhaustive coverage; the goal is a clean first cut that is inspectable by hand and still rich enough to expose mode differences.

```
hidden_dag:
  n_core in {5, 10}
  depth in {2, 5}
  n_connected_distractors in {0, 5}
repair_chain:
  n_objects in {1, 3}
  n_repair_steps in {2, 3}
  n_distractor_tools in {0, 2}
seeds:
  0..4
modes:
  standard, ptc
```

The budget is set to three times the public-solver call count.

For `hidden_dag`, the public-solver count is all generated functions plus submit. For `repair_chain`, it is inspecting every tool once, scanning every object once, moving as needed, applying every required repair stage, testing every object, and submitting.

The multiplier matters. A budget that fits only an oracle-perfect trace mostly measures whether the model guessed the intended path immediately. Undertow should allow discovery, wrong hypotheses, repeated scans, and occasional recovery while still making waste visible.

## Diagnostic results

The first run compared standard and PTC modes across the generated grid using Qwen3.6-35B-A3B and GLM-5. The normalized result set contains 307 episodes, with most cells containing 40 episodes. One PTC `repair_chain` slice for GLM-5 is incomplete and should be treated as directional.

The headline pattern was strong for PTC. On `hidden_dag`, PTC agents built the expected local machinery: a value table, a queue of callable functions, and a simple execution loop. They solved the task at high reliability. On `repair_chain`, PTC agents often completed the hidden repair transitions, but many failed the final reward because they submitted prose or an incorrect submission format.

That was a bug in the environment (has been fixed post-run), but it is not the same as failing to operate the hidden world.

The more important event was that the standard comparison exposed a harness bug. Standard mode reconstructed the environment from a serialized snapshot between native tool calls, but the snapshot preserved only generic episode bookkeeping. It did not preserve family-specific mutable hidden state, such as the known value table in `hidden_dag` or the current location and repair stages in `repair_chain`.

PTC mostly avoided this issue because its primitive tools were served through a live RPC-backed session. Standard mode did not. A standard agent could compute `v1`, then make the next call in an environment that had forgotten `v1`. It could move to `bay_0`, then apply a tool in an environment whose `current_location` had reset.

That means the pre-fix standard-vs-PTC numbers are diagnostic.

The diagnostic run is not useless just because the standard harness had a bug. In fact, it shows why looking at trajectories is good (thanks to Alex & Florian).

**First**, PTC agents appear to convert `hidden_dag` into the intended local-state problem. The best traces look less like improvised conversation and more like small executors. They parse the public spec, maintain known values, call functions when inputs are available, and submit the target.

**Second**, `repair_chain` shows a useful separation between progress and reward. Several PTC agents made the right state transitions but failed final submission formatting (has been fixed).

**Third**, the standard traces revealed a plausible behavioral contrast: standard agents often responded to uncertainty by repeating tools, refreshing observations, or trying nearby schemas rather than maintaining an explicit ledger.

### hidden_dag results, cautiously stated

On `hidden_dag`, the PTC agents did what the framework was designed to make possible. They treated the task as a dataflow execution problem.

The cleanest interpretation is:

* The local state machine for `hidden_dag` is a table of known values plus a queue of callable functions.
* PTC agents can build that state machine naturally inside the workspace.
* Standard agents may also solve the task, but they are more exposed to conversational state loss and repeated-call recovery loops.
* The pre-fix standard results should be rerun before any final mode comparison is made.

PTC reached `progress_score = 1.0` on average. That means the PTC agents generally completed the repair state transitions even though final reward stayed zero. They inspected tools, built stage maps from public inspection data, moved to the right bay, applied the stage-appropriate tool sequence, updated observation versions, and reached repaired states.

A sampled PTC trace shows the pattern plainly. The agent inspected tools, identified the stage sequence, moved to the bay, applied the full repair chain, scanned the machine at the repaired stage, and received a successful final test. Then it called:

```python
finish(answer="All critical machines have been repaired...")
```

The environment expected structured repaired object ids:

```python
finish({"repaired": ["machine_0"]})
```

So the final reward stayed zero and the failure was classified as `premature_submit`. That was a bug in the environment implementation and has since been fixed. PTC succeeded at the stateful control problem; it failed the finish contract.

In standard mode, the pre-fix `repair_chain` traces were dominated by repeated calls and violated constraints. Qwen standard runs produced 871 repeated-call failures and 341 constraint violations. GLM-5 standard runs produced 880 repeated-call failures and 666 constraint violations.

Before the restore fix, these numbers should not be interpreted as pure model failure. A representative trace showed a successful `move_to`, followed by an `apply_tool` response whose `current_location` was null. The agent was being judged inside a world that had lost movement state between calls.

PTC `repair_chain` failures were different. They were dominated by repeated calls and premature submit, with much lower constraint-violation pressure. The agent was still inefficient, but it was inefficient around a working plan rather than lost before the plan.

The diagnostic evidence fits the conditional PTC claim from above: the workspace helps when the task can be converted into a local state machine whose updates depend on external tools. In `hidden_dag`, that state machine is a value table. In `repair_chain`, it is a repair ledger. The model is not merely choosing the next action; it is building the apparatus that chooses actions.

<a id="flow-closing"></a>
## Closing

The old habit in tool-use benchmarks is to ask whether the model reached the answer. Undertow keeps that question, but it is more interested in the account book: what the agent knew, what it changed, what it repeated, what it used too early, what it forgot, and what it submitted.

Undertow gives a controlled way to generate worlds, vary difficulty, audit traces, and locate failure for programmatic tool calling.

Programmatic tool calling is interesting when the program becomes a disciplined interface to a world the model cannot inspect directly.
