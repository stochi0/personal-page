# Undertow: A Synthetic Benchmark Experiment for Programmatic Tool Calling

*A preliminary benchmark-design note about generated hidden worlds, mutable state, and when a programmable workspace might help tool-using agents.*

## TL;DR

Undertow is a small synthetic benchmark experiment built around one question:

> Can we generate hidden, stateful tool-use worlds where a programmable workspace should matter, and measure where agents fail?

The current version has two generated task families. In `hidden_dag`, the agent gets initial values, generated function schemas, and a target variable. It does not get the true value table or the solution order, so it must discover a valid execution order by calling functions and tracking returned values. In `repair_chain`, the agent gets machine ids, tool ids, bay ids, and primitive operation schemas. It does not get the repair plan, so it must inspect tools, scan machines, move between bays, apply stage-specific repairs, track stale observation versions, test repaired machines, and submit repaired ids.

The benchmark is synthetic on purpose. The goal is not realism by surface detail; it is controllability. We can vary graph size, dependency depth, distractors, number of machines, repair depth, stale state, partial observations, and budget pressure while keeping the hidden checker exact. That makes it possible to ask which parts of tool use break: planning, value propagation, state tracking, recovery, final answer shape, or budget discipline.

The current results should be read as diagnostic evidence, not as proof that programmatic tool calling wins. Some PTC traces look like the intended local state-machine strategy: a value table for `hidden_dag`, a repair ledger for `repair_chain`. But the run is small, and the first standard-vs-PTC comparison was affected by harness and final-answer-contract issues. The useful conclusion for now is that Undertow makes these failures visible and gives a controlled way to rerun the comparison.

<a id="flow-why"></a>
## Why build this

Tool-use evaluations often collapse many behaviors into one score. A model might fail because it picked the wrong tool, called a child function before its parent, forgot a value, reused a stale observation, moved the world into the right state but submitted the wrong JSON shape, or simply ran out of budget. A single success rate usually cannot distinguish those cases.

Undertow tries to make the failure surface explicit. It creates hidden environments where the agent only sees public identifiers, public tool schemas, and observations returned from primitive tools. The hidden state and reward checker remain protected. The benchmark records the trace and labels failures mechanically.

That gives the experiment a narrower purpose than "evaluate agents" in general:

* Define generated tool-use worlds with hidden structure.
* Expose only primitive public operations.
* Vary difficulty through explicit knobs.
* Score with a hidden checker.
* Audit traces by failure type, progress, and wasted calls.
* Compare ordinary tool calling with a programmable workspace only after the harness is clean.

The main hypothesis is conditional:

> A programmable workspace should help when the task can be turned into a local state machine whose updates depend on external tools.

In `hidden_dag`, that local state machine is a table of known values and pending functions. In `repair_chain`, it is a repair ledger containing machine state, tool-stage compatibility, current location, observation versions, tested machines, and final repaired ids.

That hypothesis is not yet a result. Undertow is the first pass at an environment where it can be tested.

<a id="flow-funcbenchgen"></a>
## Relationship to FuncBenchGen

FuncBenchGen is the closest reference point. It argues that synthetic function-calling benchmarks can be useful when they are contamination-resistant, generated at evaluation time, and precise about which capability they test. Its core task casts multi-step function calling as traversal over a hidden function-dependency DAG: the model receives generated functions, known input variables, and a target variable, but not the dependency graph itself. Difficulty is controlled through graph size, dependency depth, and distractor functions.

The important lesson is not that every benchmark should be synthetic. It is that a synthetic benchmark has to earn its keep by making the task formal and the knobs meaningful. If the environment is artificial, the reader should still know exactly:

* what the agent sees,
* what remains hidden,
* what action changes the world,
* what counts as success,
* which knob stresses which capability,
* and why the result should matter.

Undertow keeps that discipline but changes the object of study.

<a id="flow-setup"></a>
## Benchmark setup

An Undertow task is generated from a seed and a set of family-specific parameters. The generator creates a hidden initial state, a public task specification, primitive tools, a target condition, a hidden reward checker, and a tool-call budget.

The agent receives the public task specification. It does not receive the hidden state or the checker. Each step is a tool call. The environment updates hidden state, returns a public observation, and records whether the call produced useful state change or a failure label. The episode ends when the agent submits an answer or exhausts the budget.

The two interaction modes expose the same underlying task through different control surfaces:

* **Standard mode:** the model uses native tool calls directly through the normal conversation loop.
* **PTC mode:** the model gets a persistent Python workspace and calls the same primitive tools through a small package.

PTC mode is not an oracle. It does not expose hidden state or reward. It only gives the model a place to write code, store ledgers, parse observations, retry carefully, and construct the final answer.

<a id="flow-hidden-dag"></a>
## Task family I: hidden_dag

`hidden_dag` is the functional baseline. It asks whether an agent can execute a generated dependency graph when it sees function schemas but not the hidden value table or solution order.

The public spec includes:

* initial variable values,
* a target variable,
* generated function names,
* each function's input variables,
* each function's output variable,
* and JSON tool schemas.

A small public spec might look like this:

```json
{
  "family": "hidden_dag",
  "initial_values": {
    "v0": 343,
    "source_3": 706,
    "source_4": 478
  },
  "target": "v3",
  "functions": [
    {
      "name": "core_0",
      "inputs": ["v0"],
      "output": "v1",
      "description": "Transforms v0 into v1."
    },
    {
      "name": "core_3",
      "inputs": ["source_3"],
      "output": "v4",
      "description": "Transforms source_3 into v4."
    },
    {
      "name": "core_1",
      "inputs": ["v1", "v4"],
      "output": "v2",
      "description": "Combines v1, v4 into v2."
    }
  ]
}
```

A successful function call returns only the new observation:

```json
{
  "ok": true,
  "variable": "v1",
  "value": 596
}
```

The intended solver is simple but unforgiving:

```python
known = dict(task["initial_values"])
remaining = list(task["functions"])

while target not in known:
    progressed = False
    for fn in list(remaining):
        if all(name in known for name in fn["inputs"]):
            result = call(fn["name"], **{name: known[name] for name in fn["inputs"]})
            known[result["variable"]] = result["value"]
            remaining.remove(fn)
            progressed = True
    assert progressed

submit(value=known[target])
```

The difficulty knobs have concrete meanings:

* `n_core` increases the number of required functions and therefore the amount of value bookkeeping.
* `depth` increases the longest dependency chain and therefore the cost of one missed value.
* `n_connected_distractors` adds irrelevant functions that look locally callable because they share variables with the solution graph.
* `n_disconnected_distractors` adds irrelevant functions outside the solution path.
* `observation_mode=minimal` removes reminders of previously known values, stressing memory or local state.
* `invalid_return_mode=noisy` can pollute the ledger with fake values after bad calls.

This family is deliberately close to FuncBenchGen. It gives Undertow a known, dry accounting task before moving to mutable state. If PTC helps here, the mechanism should be legible: the model writes a value table and a callable-function queue. If standard mode solves it just as well, that is also useful information.

<a id="flow-repair-chain"></a>
## Task family II: repair_chain

`repair_chain` is the state-control family. The agent is not just computing a value; it is operating a hidden environment whose state changes after actions.

The public spec includes:

* machine ids,
* bay ids,
* available tool ids,
* the target `"repair all critical machines"`,
* and operation schemas.

It does not reveal the required repair sequence for each machine. The agent must infer enough of the hidden state through tools:

* `scan_object(object_id)` reads machine status, repair stage, observation version, and sometimes part/location.
* `inspect_tool(tool_id)` reveals compatible parts and repair stages.
* `move_to(location)` changes the agent's current bay.
* `apply_tool(object_id, tool_id, observation_version)` attempts a state-changing repair action.
* `test_object(object_id, observation_version)` checks whether a machine is repaired.
* `submit(repaired=[...])` finishes the task.

A tool inspection response can make the hidden plan inferable:

```json
{
  "ok": true,
  "tool_id": "tool_0",
  "kind": "repair",
  "compatible_parts": ["part_0", "part_1"],
  "compatible_stages_by_part": {
    "part_0": [0],
    "part_1": [0]
  }
}
```

The intended local strategy is a ledger, not a single chain of calls:

```python
stage_tool = inspect_all_tools()
machines = scan_all_machines()

for machine_id, machine in machines.items():
    move_to(machine["location"])
    while machine["status"] != "repaired":
        tool_id = stage_tool[(machine["part"], machine["repair_stage"])]
        result = apply_tool(
            object_id=machine_id,
            tool_id=tool_id,
            observation_version=machine["observation_version"],
        )
        machine["repair_stage"] = result["repair_stage"]
        machine["status"] = result["status"]
        machine["observation_version"] = result["observation_version"]
    test_object(machine_id, machine["observation_version"])

submit({"repaired": sorted(machines)})
```

The difficulty knobs map to state-tracking pressure:

* `n_objects` increases the number of machine records the agent must maintain.
* `n_repair_steps` increases the length of each mutable state transition chain.
* `n_tools` increases the search space for stage-compatible tools.
* `n_distractor_tools` adds tools that are syntactically available but never repair machines.
* `staleness_rate` makes observation versions become invalid after repair actions.
* `partial_observation_rate` hides fields such as part or location from scans.
* `shared_tool_ambiguity` makes tool compatibility less one-to-one.

This is the family that motivates Undertow. A standard tool-calling agent can solve it, but it has to carry several changing tables through the conversation loop. A PTC agent can make those tables explicit. The benchmark is meant to test whether that difference becomes measurable as state, depth, distractors, and stale observations scale.

<a id="flow-ptc-change"></a>
## What PTC changes

The PTC interface changes the control surface, not the information available to the agent.

In standard mode, the agent sees schemas, calls tools, reads observations, and chooses the next action through the normal tool-call loop. In PTC mode, the agent uses a persistent Python workspace:

```python
from ptc import task, tools, finish

spec = task.spec()
tools.list()
tools.call("scan_object", object_id="machine_0")
finish({"repaired": ["machine_0"]})
```

The same primitive tools are underneath. Hidden state stays hidden. The checker stays hidden. The advantage, if any, has to come from the agent writing durable local machinery: queues, ledgers, parsers, assertions, retry policies, and final-answer constructors.

That is why the claim should be tested conditionally:

> PTC should help when the agent can reduce a hidden tool-use task to explicit local state plus controlled external updates.

`hidden_dag` is the value-table version of that claim. `repair_chain` is the mutable-world version.

<a id="flow-metrics-grid"></a>
## Metrics and knobs

Undertow reports both final outcomes and the shape of the trajectory.

The core metrics are:

* `success`: whether the submitted answer satisfies the hidden checker.
* `num_calls`: consumed tool-call budget.
* `budget`: total allowed tool calls.
* `progress_score`: family-specific partial progress.
* `useful_calls`: calls with a non-empty state delta.
* `wasted_calls`: calls that produced a failure code.
* `failure_counter`: counts by failure type.
* `first_failure` and `final_failure`: where the trace first broke and how it ended.

The failure taxonomy is mechanical:

* `invalid_action`: the tool does not exist or the episode is closed.
* `schema_error`: arguments do not match the schema.
* `unknown_state_reference`: the agent referenced an unknown object, tool, or variable.
* `incorrect_value`: the agent supplied a known variable with the wrong value.
* `stale_state_reference`: the agent used an obsolete observation version.
* `violated_constraint`: the action was syntactically valid but violated world constraints.
* `distractor_action`: the agent used an irrelevant function or distractor tool.
* `premature_submit`: the submitted answer did not satisfy the target.
* `budget_exceeded`: the agent tried to continue after consuming its budget.
* `repeated_call`: the agent repeated an identical call.

<a id="flow-diagnostic"></a>
## Preliminary diagnostics

The current result set should not be read as the final PTC comparison. It is useful because it exposed three things the benchmark needs to measure cleanly:

* whether agents convert the public spec into the intended local state machine,
* whether failures come from hidden-world control or from interface/harness issues,
* and whether progress diverges from final reward.

The first standard-vs-PTC run also exposed a harness bug. Standard mode reconstructed the environment from a serialized snapshot between native tool calls, but the snapshot originally did not preserve all family-specific mutable hidden state. That matters directly: a standard agent could compute a value and then call the next function in an environment that had forgotten the known-value table, or move to a bay and then apply a tool in an environment whose current location had reset. That issue has since been fixed, so the pre-fix comparison should be replaced before making a strong mode claim.

There was also a final-answer-contract issue in `repair_chain`. Some PTC traces completed the repair transitions but submitted prose instead of the structured answer the checker expected:

```python
finish("All critical machines have been repaired.")
```

The checker expected:

```python
finish({"repaired": ["machine_0", "machine_1"]})
```

### hidden_dag

The main question for `hidden_dag` is whether the agent performs disciplined dataflow execution: call only functions whose inputs are known, pass exact values, store returned variables, avoid distractors, and submit only after the target is known.

Some PTC traces did exactly what the environment was designed to elicit: inspect the public spec, build a value table, maintain a pending queue, and submit the target. Standard agents can also solve this family, and in some current cells do so efficiently. That is why this family should be treated as a calibration task, not as the decisive PTC result.

### repair_chain

`repair_chain` is the more important family for Undertow's long-term purpose because it stresses hidden mutable state. The agent has to maintain a record of what is true now, not only what was observed earlier.

The diagnostic traces show why progress and success need to be separated. A run can repair machines and still fail final reward because the final answer shape is wrong. Another run can spend many calls repeating scans, violating location constraints, or applying stale versions without making much state progress. Those are different failures and should not be collapsed into the same story.

## Closing

Undertow is a synthetic benchmark experiment, not a finished benchmark suite. Its value depends on being precise about the generated task, the hidden state, the public interface, the difficulty knobs, and the failure taxonomy.

The core idea is simple: if an agent is controlling a hidden world through primitive tools, final success is only one part of the story. We should also know what the agent believed, what it changed, what it repeated, what went stale, what violated the world constraints, and whether the final answer matched the checker.

That is the point of Undertow v0: create a controllable environment where those questions can be asked cleanly, then rerun the PTC-vs-standard comparison without pretending that the first diagnostic run already settled it.
