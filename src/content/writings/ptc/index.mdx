# Programmatic Tool Calling Environments 
*A research note*

Most agent environments begin from the same arrangement: there is a task, there is a model, and there is a set of tools. The model calls those tools, observes their outputs, and eventually returns an answer.

This arrangement is useful, and it has taught us a great deal. But it also hides a more basic question. The question is not merely whether a model can use a tool. The deeper question is whether a model can decide what tool ought to exist for it to tackle the task at hand.

That is the distinction I care about.

A good agent environment should not only test selection from a fixed menu of actions. It should test whether the model can enter a programmable workspace, observe the world through narrow primitives, and construct the machinery it needs to solve the task: solvers, verifiers, caches, parsers, state trackers, search procedures, experiment loops, and local abstractions.

This is what I mean by a Programmatic Tool Calling environment, or PTC environment. The environment should expose the world, the model should construct the method, and the reward should remain outside both.

In compact form:

```text
Environment:
    exposes primitive observability

Model:
    synthesizes task-specific cognition

Reward:
    remains hidden, external, and immutable
```

That is the whole design principle.

Expose observability, not strategy.

## The Boundary


The important boundary is between observing the environment and being handed cognition by the environment.

Take a graph task. It is reasonable for the environment to expose primitive operations such as:

```python
neighbors(node)
has_edge(src, dst)
```

These are observations. They let the model inspect the hidden object.

It is a different matter to expose:

```python
bfs(start, depth)
shortest_path(src, dst)
validate_answer(answer)
```

Those are no longer merely observations. They are pieces of the solution procedure. Sometimes that is acceptable, but it measures a different thing.

In a PTC-native graph environment, I would rather expose something closer to:

```python
env.query("neighbors", node="A")
env.query("has_edge", src="A", dst="B")
finish(answer)
```

Then the model has to write its own machinery:

```python
def bfs_exact_depth(start, depth):
    ...

def cache_neighbors(node):
    ...

def verify_candidate(candidate):
    ...
```

The verifier written by the model is not the reward function. It is only a local instrument of belief. The real evaluator remains hidden.

That distinction matters.

```text
Model-written verifier:
    confidence signal

Hidden evaluator:
    reward authority
```

If the model builds a checker, that is programmatic reasoning. If the environment exposes the true grader, that is no longer the same experiment. The model may build epistemology; it should not be handed authority.

## From Tool Use To Tool Synthesis


Ordinary tool use has a simple structure:

```text
model → calls provided tool → observes result → calls another tool → answers
```

PTC should have a different structure:

```text
model → writes solver → runs experiments → builds cache
      → writes verifier → debugs procedure → submits answer
```

The model is no longer merely choosing actions. It is building an operating layer over the task.

This is the part I think is easy to miss. The workspace is not just a scratchpad. It is where the model’s external cognition lives.

A chat transcript is a poor place to store operational state. It mixes assumptions, errors, observations, plans, tool outputs, and social language into one growing object. It is readable at first, then slowly becomes a swamp.

A programmable workspace gives the model a different kind of surface. It can write files, build small libraries, cache observations, create test cases, maintain state outside the prompt, and turn repeated confusion into a reusable procedure. The REPL is only the execution surface; the workspace is the real object.

## Why GraphWalks Is A Useful First Toy, But Not The Destination


GraphWalks is a good place to begin because it is clean. There is a hidden graph, the model receives an operation, the model must return a set of nodes or a path-like answer, and the environment can expose primitive access to the graph.

I tested this direction by making a PTC-style version with primitives such as:

```python
env.query("neighbors", node=...)
env.query("has_edge", src=..., dst=...)
finish(answer)
```

The point was not that GraphWalks itself is hard for current models. It probably is not. Recent models have become very strong at small coding and algorithmic tasks, so if the graph operation is sufficiently explicit, a capable model can often write the needed traversal code without much drama. In that sense, GraphWalks is not the grand challenge.

But that is exactly why it is a useful first instrument. A simple graph environment lets us study the design boundary without too much noise.

If the environment exposes `neighbors` and `has_edge`, the model must still decide whether to write BFS, DFS, caching, deduplication, exact-depth checks, or a local verifier. If the environment exposes `bfs`, the question has already been partly answered by the benchmark designer.

So GraphWalks is not interesting because it defeats modern models. It is interesting because it gives a minimal design practice:

```text
Start with a hidden structure.
Expose only primitive observations.
Require one final answer.
Let the model construct the missing algorithmic layer.
Inspect what it builds.
```

That practice can then be lifted into harder environments. GraphWalks is the microscope, not the specimen.

## The Design Pattern


The useful pattern is simple. First, choose a hidden state: a graph, a database, a symbolic machine, a policy book, a codebase, a world model, a conversation state, or a constraint system.

Second, expose narrow observation primitives. The model may read a node, check an edge, query a record, run an experiment, inspect a tool schema, observe a transition, or read a policy clause.

Third, withhold strategy. Do not provide the solver, the planner, the true verifier, or the hidden reward.

Fourth, give the model a programmable workspace. The model should be able to create the machinery it thinks the task requires.

Fifth, evaluate both the answer and the trajectory. The answer tells us whether the model succeeded; the trajectory tells us what kind of cognition it synthesized.

This is the design practice I want to use for more complex environments. The point is not to make tools scarce for aesthetic reasons. The point is to separate the primitive interface from the reasoning procedure.

A good PTC environment should be generous enough that the model can observe the world, but restrained enough that it has to build the method.

## Single-Turn Does Not Mean Simple


At first, PTC sounds naturally multi-turn. Web agents, SWE agents, Tau-style customer service tasks, operating system tasks, and long-horizon workflows all seem to fit the paradigm naturally.

Those are important. But I think single-turn PTC environments may be the cleaner microscope.

Single-turn does not mean the model does one thing. It means there is one final submission. Inside that submission, the model may still do a great deal: inspect, write code, run experiments, cache results, form hypotheses, verify candidates, debug failures, and submit.

A single-turn GraphWalks task can still require rich internal computation. The model receives one graph operation and must return one final answer, but before that, it may inspect the hidden graph, write traversal helpers, cache neighbor calls, implement BFS, verify candidates, and only then call `finish(answer)`.

That matters because it separates two different questions:

```text
Single-turn PTC:
    can the model synthesize useful computation?

Multi-turn PTC:
    can the model preserve and revise useful computation over time?
```

The first is the microscope. The second is the stress test.

If a model cannot build useful computation in the single-turn case, adding multi-turn persistence will not make the problem more principled. It will just make the failure harder to read.

## Toward More Serious Environments


GraphWalks gives the cleanest small case, but the same pattern should scale. The more serious environments are the ones where the model must build an operating layer, not just a solver.

For long-reasoning tasks, the hidden state may be a large dependency structure. The model should build subproblem tables, constraint stores, search scripts, checkpoint files, local consistency checks, and backtracking procedures.

For Tau-style environments, the hidden state is partly the conversation, partly the user goal, partly the policy, and partly the tool state. The model should build a conversation state tracker, policy parser, tool wrappers, commitment log, pending-action checker, and completion verifier.

For a black-box symbolic environment, the hidden state may be an unknown machine. The model should build an experiment runner, query cache, transition hypotheses, sequence search, and candidate verifier.

This is why I like the idea of a Ritual Machine environment. The machine accepts sequences of symbols and emits signals. The model cannot inspect the transition table. It can only run experiments.

The interface might be:

```python
env.query("info")
env.query("try_sequence", sequence=[...])
finish(sequence)
```

The model then has to invent the experimental method. Not merely solve the puzzle, but invent the method by which the puzzle becomes solvable.

That is the kind of environment where PTC becomes more than a harness detail.

## What I Want To Measure


Final reward is necessary, but it is not sufficient. For PTC, the important evidence is in the trajectory.

I want to know whether the model writes reusable helper functions, builds solvers, creates local verifiers, uses caches, runs experiments, recovers from errors, constructs intermediate representations, keeps durable state, and checks its answer before submission.

For GraphWalks, the relevant questions are concrete. Did it write BFS? Did it cache neighbor calls? Did it verify exact-depth reachability? Did it deduplicate answers? Did its local verifier correlate with the hidden reward?

For Tau-style tasks, the question changes. Did it track commitments? Did it parse policy? Did it wrap tools safely? Did it check pending actions before execution?

For long-reasoning tasks, the evidence is different again. Did it create a dependency graph? Did it checkpoint sub-results? Did it notice contradictions? Did it backtrack?

The score says whether the model won. The trajectory says what it became in order to win.

That is the measurement I care about.

## Why This Matters For RL


For RL, PTC environments enlarge the behavioral space. A standard tool-use environment can reward the model for selecting useful actions, but a PTC environment can reward, or at least reveal, deeper intermediate behaviors: tool construction, self-verification, state management, debugging, search, abstraction, and experimentation.

These are the behaviors that seem closer to long-horizon agency.

A model that learns to call tools is useful. A model that learns to build the right tools for a task is more interesting.

The stronger hypothesis is this: advanced agents may not come only from longer internal reasoning traces. They may come from models learning to construct external executable cognition.

Solvers. Verifiers. Memory systems. Planners. State trackers. Experimental loops.

This is why the environment design matters. The environment determines what kind of behavior can appear.

If the environment gives a tool menu, the model learns to select. If the environment gives a programmable workspace with primitive observations, the model may learn to construct.

Those are not the same capability.

## The Caution


There is a cheap version of this idea: give the model a large number of tools, call it PTC, and declare the environment agentic.

That is not what I mean.

The point is not tool abundance. The point is the boundary. A good PTC environment should be strict about three things:

```text
1. Primitive observability:
       the model can inspect the world

2. Model-side cognition:
       the model writes the strategy

3. Hidden reward:
       the environment evaluates externally
```

If the environment gives away the solver, the benchmark is measuring tool use. If the environment exposes the grader, the benchmark is measuring reward access. If the workspace becomes an unstructured diary, persistence may become ceremony.

The question is not whether the model created files. The question is whether the created files changed the next action.

The question is not whether the model wrote code. The question is whether the code became a useful cognitive instrument.

That is the standard I want these environments to meet.

## Current Direction


The immediate plan is to use GraphWalks as the small design probe, not as the final claim.

GraphWalks is useful because it lets me test the basic PTC interface:

```text
hidden state
primitive queries
model-written solver
model-written verifier
external evaluator
trajectory inspection
```

Once that pattern is clean, the same design can be used for richer environments:

```text
GraphWalks:
    hidden graph reasoning

LongCoT-PTC:
    executable long-horizon reasoning

Tau-style PTC:
    programmable agent infrastructure

Ritual Machine:
    black-box symbolic system identification
```

The goal is not to produce a large benchmark immediately. The goal is to develop a reliable practice for constructing environments where the model has to build the missing cognitive layer.

That is the thing I want to make legible.

## Final Word


The next interesting RL environments for agents should not only test whether models can use tools. They should test whether models can build tools.

Not whether the model can call `bfs`, but whether it can discover that a traversal procedure is needed, write it, cache around it, verify with it, and submit the result.

Not whether the model can call a customer-service API, but whether it can build a policy checker, a state tracker, and a safe action layer around the API.

Not whether the model can produce a long chain of thought, but whether it can turn the reasoning process into durable executable cognition.

That is the point of PTC environments. The model should not merely operate inside the environment; it should slowly construct the little machine by which the environment becomes solvable.

That is the capability boundary worth measuring.
